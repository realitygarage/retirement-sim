User asked for a reclassification pass: split the 10 items into four buckets -- (A) modeling

own fallback to it never fired, since wfData always covers live) -- it only mattered for
pinned-scenario comparison lines, tying A4 directly to A2. cashAst was confirmed empirically
inert (exactly $0 across all 21 rows, all 3 pins) -- the "shortfall exceeds proceeds" branch it
exists for never fires given current pooled-routing design.

--- PHASE 1: IMPLEMENTATION ---

engine.js additions (buildScenario's ~830-line body deleted entirely, replaced with):
- computeDispositions(p): extracted from inside buildScenario -- the property-disposition /
  CA-cap / obligation-offset / same-year-bump computation, now a standalone pure function of
  params only. Needed so the monthly engine's sale-year pooled routing no longer depends on
  buildScenario's output (which would have been circular once buildScenario is gone).
- buildMonthlyScenario(p): the single engine. Built from wfData's logic, ported to a pure
  function taking the full params object directly (turns out liveParams already carried every
  Cash-Flow-tab field wfData used to read from separate component-scoped variables -- no
  separate "cfSettings" bundle needed). Implements A1 (maintenance folded into tier1, uncapped,
  inflated, res6/res15/resLaf retired entirely, sale-year pooled routing narrowed from 5
  reserve/buffer buckets to 2 (rd/ob only)); adds D1 (continuous per-property appreciation
  tracking, using true mo/12 instead of buildScenario's frozen once-per-row elapsedYrs); adds
  D2 (per-debt ccBal/sophiaBal/nolanBal exposure, generalized per-property mortgage/rental
  breakdown including the previously-missing Lafayette balance); adds D3 (totalOut assembled
  from tier1+sweep+mtgExtra, deliberately excluding rd/ob top-ups since those are discretionary
  and never compete with "does income cover costs"); adds D4 (structured loan/mortgage event
  objects alongside the existing display strings).
- aggregateMonthlyToAnnual(wfRows, p): the annual/Simulator view, now pure rollup. Stocks =
  first-of-calendar-year snapshot (pre-decrement, matching the existing v4.3.0 convention
  automatically since wfData's own rows already carry this snapshot property); flows = summed
  over the year's real months. Produces the same field names buildScenario used to, so most of
  App.jsx's UI layer needed zero changes.

App.jsx: wfData now calls buildMonthlyScenario(liveParams) directly; liveRows is now
aggregateMonthlyToAnnual(wfData, liveParams) -- reordered so wfData computes first (it no
longer depends on liveRows the way the old sale-year routing did). buildRowsFromSnapshot (pins)
now returns {rows, wfRows} and runs the full monthly engine per A2. addPin simplified to reuse
live's already-computed rows/wfData instead of a redundant second buildScenario call.
chartData's fcfChart/sweepChart usage removed per A4 -- pin_<id>_di/sweep now read the pin's
own aggregated surplus/debtSweep/sweepToSavings fields, sourced identically to live. Fixed the
dev-mode conservation audit and the pooled-routing display line to include the new
wfReserveFill term (B2's fix: reserve top-ups and true sweep are no longer conflated).
window.__engine no longer exposes buildScenario; exposes buildMonthlyScenario/
aggregateMonthlyToAnnual/computeDispositions instead.

Five bugs found and fixed during my OWN implementation, before ever showing results:
1. Hardcoded 252-month loop bound only equals 21 calendar years when startMonth is January --
   for the actual default (July) it overshoots into a spurious 22nd partial year. This was
   ALREADY a latent bug in the pre-v5 wfData block too; it just never surfaced because the old
   chartData only ever visited years present in the separately-bounded 21-row buildScenario
   output. Fixed via the existing monthsElapsedBeforeYear(21, startMonth) helper.
2. First draft of aggregateMonthlyToAnnual forgot to fold savingsAcc into nw at all -- caught
   because it made nwYr10 suspiciously identical across all 3 scenarios.
3. The serious one: propMtgBal wasn't gated by ownership, so a sold property's mortgage state
   (which simply stops being stepped once sold, staying frozen at its pre-sale balance) was
   being read unconditionally -- wrongly subtracting a paid-off $805K mortgage from NW for
   every year after 6th St's 2027 sale. This is what produced the alarmingly large NW deltas in
   my first smoke test.
4. famLoanBal missing a /1000 (reporting $21,432K instead of $21K).
5. cumDraw dropped the one-time settlement draw -- the old engine blended it into the same
   drawInc field before computing cumDraw; the new drawInc only covers scheduled ad-hoc draws.

--- THE TWO CATCHES (both legitimate, both changed the reported numbers) ---

Catch 1: I reported "A1's real effect is tiny (+$7-10K cumMaint), NW actually goes UP not
down, the opposite of the Phase 0 estimate" as a "correction." User pushed back, asking for a
full per-scenario NW reconciliation (named, sized line items summing to the total, no
"baskets") and an explanation of a savingsAcc-in-nw discrepancy I'd mentioned in passing. Built
real counterfactual engine variants (not formulas on paper) to isolate each factor's marginal
NW contribution. This process itself surfaced that my "correction" was ALSO wrong: I had only
isolated A1's direct cost-line basis swap (small), missing that A1 has a second, dominant
mechanism -- the old capped reserve system redirected its ENTIRE ongoing maintenance amount
into the savings sweep once each reserve filled (~2028-2031), and removing the cap removes
that redirect too. Building the full counterfactual (which reintroduces the old res6/res15/
resLaf/maintRedirect mechanism) found a real double-counting bug in my own analysis script
(maintenance was being subtracted twice -- once via an uncapped tier1 line, again via a capped
reserve accrual -- something the real old dual-engine architecture never did, since those were
two fully separate, non-interacting computations). Fixed, then validated: A1's true full effect
is -$197K to -$267K at year 20, matching the ORIGINAL Phase 0 direction and within range of the
original magnitude -- the "correction" had been the wrong one, not the original estimate.

Also answered directly: was the user's actual, currently-committed, pre-refactor displayed NW
ever wrong? No -- checked precisely. Live scenarios' displayed NW already included savingsAcc
correctly (added back in at the old chartData layer). Pinned comparison lines used a cruder
annual-compounding approximation instead of true monthly compounding, but measured against the
user's actual 3 saved pins, that approximation was only off by $8-44K (1-4%) -- nowhere near
the six-figure swings that came from comparing against the never-displayed internal liveRows.nw
field. The user's past decisions based on displayed numbers were not meaningfully wrong.

User also asked for an audit of whether any OTHER 0a prediction was similarly mis-framed.
Found: B1's predicted "debtClearYr may shift up to ~1yr" didn't materialize for ANY of the 3
tested pins (mechanism is real, just never straddles a year boundary in these scenarios) -- and
a genuinely new, previously-unflagged finding: maxDI (a keyStats headline figure) swings 75-85%
lower under the new disc-based surplus definition, which I hadn't attached a magnitude to in
Phase 0.

Catch 2 (smaller): before approving commit, user asked (1) for maxDI's old vs. new definitions
side by side with what decision each supports, and (2) to re-rank the 3 pinned scenarios on the
corrected NW to see if the honest maintenance treatment changes which scenario looks best.
Investigating (1) found maxDI is not read anywhere in the current UI at all -- a live

is unchanged at both year 10 and year 20 for all 3 pins (margins compress, order doesn't flip),
though workFreeYr's ranking did flip for one scenario (pin_37, last -> tied-first).
