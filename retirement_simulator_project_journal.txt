--- WHAT THIS PROJECT IS ---

A personal retirement financial planning simulator modeling income streams,
debt paydown, real estate equity, and net worth projections over a 21-year
horizon (2026-2046). Built for Bob's personal retirement planning -- not a
commercial product. Models a complex household financial picture: pension,
Social Security (self + spouse Brenda), rental income from multiple
properties (6th St primary residence, 15th St duplex, Lafayette property),
high-interest debt paydown (credit cards + family loans ~$306K total at
launch), and work income tapering toward zero ("work-free" target).

--- TECH STACK ---
React + Vite + JSX, multi-file architecture (refactored from a single
3700+ line file in Session 11).
  src/engine.js          -- pure calculation functions, no React
  src/defaults.js        -- default scenario values, params builder
  src/relationships-data.js -- Input/Output Map diagram data
  src/App.jsx            -- all React state, hooks, UI, charts

Playwright test suite (retirement-simulator.spec.js, project root) is the
quality gate -- every change should be followed by a test run.

--- ARCHITECTURE / ENGINE NOTES ---

Two parallel calculation engines exist by design:
  1. ANNUAL ENGINE (engine.js: buildScenario) -- one row per year, used for
     most charts (NW, debt balance, income/cost breakdowns). Faster, used
     for most chart series.
  2. MONTHLY ENGINE (wfData, lives inside App.jsx) -- one row per month,
     used for the Cash Flow tab waterfall and as the authoritative source
     for sweep-to-savings timing (savingsAcc). More precise on exact months
     (e.g. exactly when HI debt clears), but NOT exposed via engine.js.

IMPORTANT LESSON (Session 11): Don't try to replace wfData's savingsAcc
with an annual-engine approximation -- this was attempted and caused NW
to under-report by ~$500K because the annual engine loses precision on
the exact month debt clears and sweep starts. The monthly wfData engine
is correct; chartData blends annual engine NW + wfData savingsAcc.

Components that must stay inside App.jsx (do not extract):
  Chart, WorkCurveEditor, BREAKDOWNS -- all close over ~40 App-local
  variables (state, setters, computed values). Extracting them would
  require extensive prop-drilling. Not worth it given current scale.

Key naming/units conventions:
  - Dollar values often stored in $K (thousands) in engine rows (nw, hiDebtK,
    reEquity, etc.) but $M (millions) in chartData points for NW chart
    (matches yFmt on the Y axis)
  - savingsAccK = $K, used in NW breakdown table
  - liqNW = $M, used by the book/liq toggle

Common test-writing gotchas learned:
  - Use native React setter pattern for sliders in tests:
    `Object.getOwnPropertyDescriptor(HTMLInputElement.prototype,'value').set`
    then dispatch both 'input' and 'change' events with bubbles:true --
    setting el.value directly does NOT trigger React's onChange.
  - Prefer `text=` locators over element-type-specific ones (span, div) for
    resilience against markup changes.
  - When parsing currency strings with M/K suffixes in assertions, always
    use parseFloat with a decimal-preserving regex, not parseInt.

--- DEVELOPMENT WORKFLOW (established practices) ---

Two-tool workflow:
  1. claude.ai (chat interface) -- for design discussions, screenshot
     review, feature planning, journal writing. Best when you want to think
     through a feature before coding it, or review test results/behavior.
  2. Claude Code (terminal, via Git Bash) -- for actual file edits, running
     tests, multi-file coordinated changes. Reads/writes files directly on
     disk, no upload/download cycle needed.

Recommended session pattern:
  1. (Optional) Design discussion in claude.ai with journal + screenshots
  2. Open Git Bash: cd /c/Users/rober/retirement-sim && claude
  3. Give Claude Code the journal + task, let it read source files directly
  4. After changes: have it run run_tests.ps1 and report results
  5. Have it commit to git: git add -A && git commit -m "v2.X.X -- desc"
  6. CRITICAL: before ending session, ask Claude Code to write the next
     session's journal file -- this is the safety net if the window closes
     unexpectedly (this happened once already, Session 12 had to be
     reconstructed from source code alone)

Git setup (initialized Session 12-13):
  - core.autocrlf set to false
  - .gitattributes enforces LF for JS/JSX/JSON/MD/TXT, CRLF for .ps1
  - This prevents noisy diffs and patch failures from line-ending drift

Version bumping rule (Bob's standing instruction): ALWAYS bump the patch
version number for every change to App.jsx, no matter how small. The
version badge in the UI header is the confirmation signal that Vite has
loaded the new file after a deploy + hard refresh (Ctrl+Shift+R).

Deployment gotcha: files must go in src/, NOT the project root. Copying to
the wrong directory is a recurring mistake -- Vite won't pick up the change
and the version badge will stay stale, which is the tell-tale symptom.
