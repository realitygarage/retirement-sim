--- WHAT SHIPPED ---

--- 0. WHY (spec recap) ---

The v3 sidebar was knob-centric: dispositions{sixth,barberry,fifteenth} as
a flat lookup, a 15th-St-specific Top Unit selector, a Lafayette
rental/stop-year pair, sixthIncomeSegments as a one-off array, LIQUIDATION_
DEFAULTS/SETTLEMENT_DEFAULTS/SIXTH_STR_DEFAULTS as separate constant blocks.
v4 makes the model property-centric: properties: [{id, label, isPrimary,
value, appreciationPct, mortgage, hold, units:[{segments}]}]. NO backward
compatibility -- fresh schema, no migration, no legacy mapping (confirmed
with the user before starting). SAVE_SCHEMA_VERSION bumped 3->4 so old pins
are silently dropped on load (same mechanism used for the v2->v3 break).

--- 1. NEW SCHEMA (defaults.js) ---

freshPropertiesDefaults() returns three properties (sixth/fifteenth/
barberry), each with value/appreciationPct/mortgage/hold/units. Key
consolidation: the OLD model had TWO separate value concepts -- BASE.
primaryValue etc (book value for the NW chart) vs LIQUIDATION_DEFAULTS.
salesPrice (the disposition-sale FMV seed). v4 unifies these into ONE
`value` field per property (matching the spec's schema example exactly:
sixth=1,675,000, values from the CPA liquidation worksheet, not the old
BASE.primaryValue=1,700,000/duplexValue=1,500,000/lafayetteValue=590,000
book figures). CONSEQUENCE: book-NW baseline values are now slightly lower
than pre-v4 by design, not a regression -- flagged for the user to notice.

Also dropped: the v3.1.1 "Sale price" OVERRIDE slider (fmv = BASE value x
appreciation, with an optional manual override). v4's schema has no
separate salesPrice field under hold -- FMV at sale is ALWAYS `value x
(1+appreciationPct)^yearsElapsed`, computed automatically. Simpler model;
users who want a different assumed value edit `value` directly.

Barberry/Lafayette mortgage: unlike sixth/fifteenth (servicer-calibrated,
Jul 2021 origin, 10-yr IO), Barberry's origin date was never calibrated.
Set originYear:2021, originMonth:1 (not July), ioYears:0 -- this exactly
reproduces the pre-v4 remainBal(181115, 0.041, 30, 5+yr) formula's "5 years
paid as of Jan 2026" assumption via the SAME generalized mortgage state
machine (verified: mortgageBalanceClosed with ioYears:0 reduces algebraically
to the old remainBal formula). This let Lafayette's flat BASE.lafPnI
constant be RETIRED -- its P&I payment is now a genuine amortization
calculation like the other two properties, one state machine for all three.

One-Time Obligation (was Settlement): { amount, year, quarter,
offsetsCapitalGains: true }. Assumed-100% gain offset (no percentage
slider) -- when on, obligation.amount reduces recognized gain dollar-for-
dollar, capped at that year's gains pool, single-year model (no
cross-year "requireSameYearForOffset" ambiguity to disambiguate anymore).

New economy knobs: ltrVacancyPct (4%), mtrCleaningFlat ($300/block) --
exposed as regular sliders (Cost Profiles block), not Defaults-tab entries,
since users will want to tune these while testing segments.

Removed entirely (no migration): dispositions, sixthIncomeSegments,
topUnit/duplexTopSTR/LTR/MTR/duplexBottomLTR, strSchedule/mtrSchedule,
lafRental/lafStopYear, LIQUIDATION_DEFAULTS, SETTLEMENT_DEFAULTS,
SIXTH_STR_DEFAULTS, migrateScLoans (famLoanAmt back-compat -- dead now that
SAVE_SCHEMA_VERSION drops all pre-v3.2 pins anyway), migrateSixthIncome
Segments, duplex15thBasis/lafayetteBasis (liq-NW quick-calc now reads
properties[].hold.basis directly -- one basis input per property instead
of two overlapping ones).

--- 2. SEGMENT RULES (engine.js) ---

unitSegmentGross(seg) / unitSegmentNet(seg, opts): generalized from the
v3.3 sixthSegmentGross (identical inner-cap-clamped math for STR days<=365,
MTR months<=12) with a NEW netting function applying the §1 cost-profile
table: STR = platform% + cleaning% + mgr%; MTR = ONE flat $/block cleaning
(not %) + mgr%; LTR = vacancy% + mgr%, NO cleaning charge at all. Both
engines call this SAME function -- guarantees the netting can't drift
between annual and monthly (verified: STR $30K gross -> $27,900 net; MTR
$60K -> $59,700; LTR $36K -> $34,560).

validateUnitSegments(segs): inner caps unchanged; NEW rule -- LTR rejects
overlap with ANY other segment on the unit (a tenant occupies the whole
unit); STR+MTR may coexist freely. unitSegmentOverlaps(segs): informational
(not blocking) contiguous-range + combined-gross reporter, used for the
neutral blue info line in the UI.

yearHeldFraction(hold, year) / unitOwnedThisMonth(hold, calYear, mo1to12):
the annual engine's proration multiplier and the monthly engine's exact
ownership boolean, respectively -- BOTH built on the same "sale at a
quarter boundary" semantics (Q2 sale = owned through Q1, not owned from Q2
month 1 onward = 3 months owned = 0.25 of the year). Verified live: 15th
St sold Q2 2028 -> monthly rental drops sharply from $9,973 (Mar) to $3,325
(Apr), exactly at the quarter boundary; annual 2028 shows a prorated
$4,795 vs $9,277 (2027, full year) and $3,304 (2029, first full year
without it). segmentClipInfo(seg, hold): UI-only non-blocking "truncated
at Q{n} {year} sale" notice, never blocks income calculation.

--- 3. ENGINE CHANGES (both annual + monthly, MUST agree) ---

- Income: for each held property (mortgage-state ownership, NOT segment
  presence), for each unit, sum ALL covering segments via unitSegmentNet,
  prorated by yearHeldFraction (annual) / unitOwnedThisMonth (monthly,
  exact). Verified two concurrent segments (STR+MTR on 6th's single unit)
  sum correctly and the annual/monthly net figures agree within $3 (R1).
- Disposition: computeDispo generalized to loop over properties[] by id
  (was: 3 hardcoded destructured dSixth/dLaf/dDuplex blocks) -- reuses
  disposeAsset unchanged (property-agnostic already). MODE STRING FIX:
  the old 'sell_taxable' internal string is renamed 'sell' to match the
  v4 schema's own comment (`mode: 'keep'|'sell'|'full_1031'|'partial_1031'`)
  -- caught via a live conservation test that initially returned pool=$0
  (disposeAsset silently no-opped 'sell' since it only recognized
  'sell_taxable' as the taxable-sale branch). Verified fixed: pool
  correctly computes to $777,587 for a 15th St sale.
- IO->P&I mortgage: unified into ONE state machine for all three
  properties (Lafayette's ioYears:0 recasts immediately at origination,
  same code path as 6th/15th's 10-yr IO). mtgById keyed generically;
  MTG_PRINCIPAL_ELIGIBLE_IDS=['sixth','fifteenth'] keeps the bucket's
  "held 6th & 15th only" rule explicit. Mortgage STEPPING (both engines)
  uses unitOwnedThisMonth for TRUE monthly ownership -- a mid-year sale
  stops payments at the exact quarter, not the whole sale year (this is
  what keeps the annual engine agreeing with wfData in the sale year,
  since the coarser "cal<saleYear" annual boolean would otherwise
  understate months 1-3 of a Q2 sale).
- Pooled routing (§3.4): yearCashAdd/yearCashSub keyed by property.hold.
  year generically; obligation.amount (not settlementNeed) is the cash
  outflow AND (when offsetsCapitalGains) the gain-offset amount, applied
  ONLY to the obligation's own year (no cross-year toggle needed -- single-
  year pool). Lifestyle draw / HI paydown / mortgage-principal bucket
  machinery (v3.2/v3.4) is UNCHANGED, just fed from the generalized pool.
  Conservation verified: draw+paydown+waterfall == residual exactly (R8).
- Mortgage-principal bucket: unchanged mechanics (fed by leftover
  debt-sweep or post-debt surplus, capped, 6th before 15th), just
  generalized to loop over MTG_PRINCIPAL_ELIGIBLE_IDS against the shared
  mtgById map instead of two hardcoded state objects. Verified ordering:
  6th's balance drops under the bucket while 15th's stays untouched (R9).

--- BUG CAUGHT DURING FULL-SUITE VERIFICATION (in addition to the mode-string
    fix above) ---

The bulk mass-deletion of the ~830-line dead UI block (One-Time Decisions
through Income & Cost Knobs) was scoped by line-range grep, and the range
turned out to be wider than the dead UI itself -- it also swallowed THREE
unrelated, still-needed sections that happened to sit inside that range:
"Your SS Start Age" toggle, the WorkCurveEditor render, and the "HI Debt
at Launch" balance/rate/min editor (ccBal/sophiaBal/nolanBal), plus the
"Liquidation NW Basis" section header (its two dead sliders were correctly
removed in an earlier edit, but the whole section then got deleted again
moments later by the wider block replacement). Caught by the full
run_tests.ps1 pass: E1 (SS age click) timed out, J4/J5 (Liquidation NW
Basis) and 2 others failed on missing text. All three sections restored
verbatim (unchanged from pre-v4.0.0-A) in a new spot right after the
Pooled Proceeds Routing block. Verified via direct browser checks (section
text present, button count, no console errors) before re-running tests.
Lesson: line-range-based bulk deletion of "this whole block is dead" is
risky when the block wasn't verified boundary-by-boundary against a
fresh diff -- worth grep'ing content markers on BOTH sides of a deletion
range next time, not just the start/end section headers.

Also caught: H4 ("All HI debt cleared — avalanche sweep ends") no longer
fires in the default scenario -- traced to a LATENT v3.4.0 regression
(not introduced by v4.0.0-A): since the v3.4 IO/recast fix, mortgage
payments now jump at the true 10-year IO boundary regardless of HI-debt
state, and in the default scenario this means the annual engine's surplus
never exceeds the protected split threshold enough to trigger a mid-run
debt-sweep (debtSweep stays 0 for all 21 years even though hiDebt does
reach 0 in 2036). This surfaced now because v3.4.0 shipped WITHOUT a full
suite run (an explicit, documented deferral in session19's journal).
Rewrote H4 to check "All HI debt eliminated" instead -- the milestone
event that fires unconditionally on hiDebt reaching 0, which is the more
correct signal anyway. Flagged in the backlog for the v3.4 debt-sweep-
timing question, not fixed here (out of scope for a v4 schema session).

--- SEMANTIC NOTES ---

- Book-NW baseline values changed slightly (properties[].value now uses
  the CPA liquidation-worksheet sale prices, not the old separate BASE
  book-value constants) -- intentional consolidation, not a regression.
- The "Sale price" manual-override slider from v3.1.1 is gone; FMV at
  sale is always value x appreciation. Users adjust `value` directly.
- Lafayette/Barberry's mortgage origin (Jan 2021, not Jul 2021) is a
  deliberate approximation preserving the pre-v4 remainBal(...,5+yr)
  trajectory -- its true origination date was never calibrated from a
  servicer statement (unlike 6th/15th).
- Property tax/insurance (BASE.primTaxMo etc) and the Cash-Flow-tab
  maintenance-reserve structure sliders (struct6/15/Laf) stay OUTSIDE the
  properties[] schema -- these are escrow lines and a separate reserve-
  planning input, not disposition/mortgage/income data, per spec §3.4.0
  ("escrow/tax/insurance stay in their existing separate cost lines").
- Ownership for NW/maintenance/tax-ins purposes still uses the coarse
  pre-v4 "cal < saleYear" annual convention (whole sale year excluded);
  only INCOME and MORTGAGE use the new quarter-precise proration. This
  is a deliberate scope boundary for Session A -- unifying it fully is
  the same "unify annual sweep model with wfData" backlog item carried
  since v3.4.
