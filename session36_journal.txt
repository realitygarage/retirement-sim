--- ITEM 1: "TOTAL WORK INCOME REQUIRED / MO" CHART -- ANNUAL RESOLUTION ---

Display-density-only fix, no engine change. The chart's `chartData` already
carries one point per calendar year (`liveRows` is one row per year from
`aggregateMonthlyToAnnual`) -- the "every 2 years" look was Recharts' own
XAxis default (`interval:'preserveEnd'`, confirmed by reading
`node_modules/recharts/.../axisSelectors.js`), which auto-skips ticks under
`minTickGap` collision pressure, not a data-sampling gap.

Added an `annualTicks` prop to the shared `Chart` component (App.jsx):
when true, `<XAxis interval={0}>` forces every year's tick, and the
breakdown-table row filter (`bdRows`, previously a hardcoded "every other
year" `i%2===0` sample used by every chart's breakdown panel) skips that
sampling and shows every row instead. Scoped to just the reqWork chart
(`annualTicks={true}` passed only at that one `<Chart>` call site) --
every other chart (FCF, Debt Balances, Net Worth, Fixed Costs) keeps its
existing biennial breakdown-table sampling, since only this one chart was
in scope.

--- ITEM 2: MONTH-BY-MONTH CASH FLOW TABLE RESTRUCTURE + WORK REQ'D COLUMN ---

Started as a narrower ask ("diagnose whether Work/Total In reflect actual
vs required work income, then add a Total In breakdown button"). Diagnosis:
Work column (`r.workIncome`) is genuinely actual/planned income from the
work curve, never `reqWork` -- correct as-is. But diagnosis surfaced a real
issue: Total In (`totalInc`) already NETS rental operating cost out of
income (`totalInc = ...+wkInc-rentalOpCost+...`), which a same-scoped
breakdown button couldn't itemize honestly (Pension+SS+Rental-gross+Work
doesn't sum to a Total In that's already down by rentalOpCost). The user
expanded the ask into a full restructure rather than paper over it.

A. RENTAL OP COST: INCOME-SIDE NETTING -> COST-SIDE LINE ITEM.
   Deliberately a DISPLAY-ONLY reclassification -- did not touch
   `buildMonthlyScenario`'s internal `totalInc`/`tier1`/`available`/
   `totalOut` formulas at all, so Free Cash (`disc`) is provably unchanged
   (R25 reconstructs `tier1` from its known fixed-cost components and
   confirms rentalOpCost is NOT in it). In the Month-by-Month table
   (App.jsx) only:
   - "Rental" -> "Rental (gross)" (no value change).
   - "Total In" cell and its breakdown total are now summed from the SAME
     already-rounded display fields (`pension+yourSs+brendaSs+rental+
     workIncome+drawInc`), not read off the engine's separately-rounded
     `r.totalInc` -- guarantees "breakdown sums to total" exactly, not
     just to the nearest rounding unit (R26).
   - "Fixed" -> "Costs"; "fixed breakdown" button -> "costs breakdown",
     which now ALSO itemizes Maint and IRMAA (previously silently missing
     from that breakdown even before this session -- tier1 has 9 terms,
     the old breakdown only listed 7) plus the new Op Cost line, so the
     10 itemized columns sum exactly to the displayed Costs total (R27).
   - New "total in breakdown" button, itemizing Pension/SS/Rental-gross/
     Work/Draw. First pass (see below) reused the always-visible leading
     columns and skipped adding new ones when nothing to show -- wrong,
     fixed in the follow-up round.

B. MONTHLY-RESOLUTION REQUIRED WORK INCOME + SHORTFALL HIGHLIGHT.
   `reqWork` (required work income) previously existed ONLY in the annual
   aggregation (`aggregateMonthlyToAnnual`), as `max(0, totalOutAvg-passive)`
   -- an annual-AVERAGE-then-floor. Added `reqWorkMo` to the monthly engine
   (`buildMonthlyScenario`): the same totalOut-minus-passive subtraction,
   but floored EACH MONTH first. These aren't interchangeable in general --
   averaging before flooring understates the gap in any year with a
   mid-year passive-income jump (Bob's SS starts in his Oct birth month,
   not Jan 1 -- exactly this case). Rather than ship two formulas that only
   coincidentally agree most years, redefined the annual `reqWork` to
   derive FROM `reqWorkMo` (`avgMo('reqWorkMo')`) instead of its own
   separate formula -- the monthly table and the "Work Income Required"
   chart now agree by construction (R28), not by coincidence. New "Work
   Req'd" column next to "Work"; red/bold shortfall highlight (+
   `data-testid="work-shortfall"`) when Work < Work Req'd (R29).

--- UI BUG FOUND IN REVIEW, FIXED SAME SESSION ---

User caught two problems after the first pass:
1. "total in breakdown" button visibly did nothing. Root cause: the first
   implementation assumed Pension/SS/Rental/Work "already exist in the
   table" (true) so only added a conditional Draw column -- but Draw is
   gated on `anyDrawInc` (any month's `drawInc>0`), and this household's
   default scenario has zero scheduled lifestyle draws, so the ONE
   conditional column never appeared. Fixed by making the breakdown
   genuinely itemize Pension/SS/Rental(gross)/Work as new columns
   (duplicating the always-visible leading columns, same precedent as
   "Maint" appearing both standalone as "Maint Res" and inside the Costs
   breakdown) plus Draw only when used -- toggling now always visibly adds
   columns.
2. Costs breakdown text was illegible (screenshot: `#f8717199`/`#34d39999`,
   i.e. the same red/green at 60% alpha, washed out against the `#0d1117`
   table background). Brightened both breakdown panels to fully-opaque
   light shades -- `#fca5a5` (Costs) and `#6ee7b7` (Total In) -- clearly
   readable while still visually distinct (italic, smaller) from the bold
   main-row totals.

R30 was rewritten to actually guard against the reported bug instead of
just clicking the button with no assertion: counts `<th>` elements matching
"SS" before/after toggling `mbm-inc-breakdown-toggle` and asserts the count
increases (previously a no-op button would have passed the old version of
this test, since it only checked the button was clickable).
