--- THE v4.2.5 BUG: DIAGNOSIS BEFORE FIX (per user's explicit process) ---

User asked for a diagnose-first, fix-after-approval workflow, correcting an
earlier framing: the design intent (from the original v4.0.0-A spec) is
that a property's SALE PRICE is a manual entry used verbatim -- appreciation
must never touch it.

Traced `disposeAsset()` (engine.js) -- clean, just uses whatever `fmv` it's
given. The bug was one level up, in its only caller `computeDispo()` inside
`buildScenario`:

    const yrIdx = Math.max(0, (hold.year||2055) - BASE.startYear);
    const appPct = prop.appreciationPct ?? p.reAppreciation;
    const fmv = prop.value * Math.pow(1+appPct, yrIdx);

`yrIdx` only looked at `hold.year`, ignoring `hold.quarter` entirely -- so
2026-Q4 -> 2027-Q1 (a 3-month move) jumped `yrIdx` 0->1, applying a full
year of appreciation. Confirmed the magnitude analytically before touching
code: $1,675,000 default value x 4% appreciation = $67,000 gross delta;
after the 6% `sellingCostsPct` haircut, $67,000 x 0.94 = $62,980 -- matches
the user's reported ~$63K to the dollar. Also confirmed 6th St's mortgage
payoff is IDENTICAL at both dates (both fall inside its 2021-2031 IO
window, where balance is flat regardless of month), so 100% of the delta
traced to appreciation, 0% to the mortgage-timing part of the report (which
was separately confirmed to be already quarter-correct via
`mortgageMonthsSince(..., quarterStartMonth(hold.quarter))`).

Also found: this is a single shared `computeDispo()` applied via
`properties.map(...)`, so all three properties (sixth/fifteenth/barberry)
had the identical bug, not something per-property. And: there is no
separate "sale price" field anywhere in the schema/UI -- `prop.value` (the
"Property value" slider) is the only entry point, doing double duty
(appreciating market value for NW while held; was ALSO being appreciated
for sale price, which is the part that's wrong). The original v4 spec
(`v4_property_centric_spec.md`) actually documents `appreciationPct` as
intended for exactly this projection, so the code matched that document --
just not the corrected design intent the user gave this session.

--- THE FIX (approved, then implemented) ---

engine.js `computeDispo()`: dropped the `Math.pow(1+appPct, yrIdx)` factor
entirely -- `fmv` is now `prop.value`, used verbatim. Removed the now-dead
`yrIdx`/`appPct` locals.

Left untouched (confirmed correct, separate code path): the NW-for-held-
properties calc a bit further down in `buildScenario`,

    const appPct = prop.appreciationPct ?? p.reAppreciation;
    valById[prop.id] = keepMap[prop.id] ? prop.value*Math.pow(1+appPct,yr) : 0;

-- this is gated on `keepMap[prop.id]` (still-held properties only) and is
the one place appreciation SHOULD compound `prop.value`, for Net Worth
tracking. Added a one-line comment at both sites (engine.js) cross-
referencing the split, plus the invariant in CLAUDE.md's "Property-centric
schema (v4)" section, so a future session doesn't reintroduce this by
"fixing" the wrong line.

UI legibility (App.jsx, per-property "Proceeds summary" block): added a new
line, `Sale price (entered — no appreciation applied): $XXXK`, sourced from
`engineRes.grossPrice` -- there wasn't a gross-price line in that block
before.

New test: R12 (`retirement-simulator.spec.js`, Group R) -- runs
`buildScenario`/`makeParams` for each of sixth/fifteenth/barberry across
three sale timings (2026-Q4, 2027-Q1 [the year-boundary case that exposed
the bug], 2040-Q2) and asserts `dispoResults[id].grossPrice` is identical
across all three. Confirmed passing with the actual invariant values:
sixth=[1675000,1675000,1675000], fifteenth=[1375000,1375000,1375000],
barberry=[625000,625000,625000].

--- v4.2.6: ITEMIZED DISPOSITION BREAKDOWN SUB-CARD ---

Follow-on request after v4.2.5 shipped: since `prop.value` now visibly does
double duty (verbatim sale price at disposition; appreciating market value
for NW while held), the user wanted the full sale->tax->proceeds derivation
made legible per property, not just the compact "Proceeds summary" line --
a collapsible sub-card, collapsed by default, itemizing every step. Hard
requirement: display-only, every value pulled directly from disposeAsset's
return object (`liveRows.dispoResults[id]`) or its pre-offset snapshot
(`liveRows.dispoResultsNoOffset[id]`) -- no tax/business logic recomputed
in the UI.

Implementation (App.jsx, all within the existing properties.map render):
- New `dispoRow(label, val, kind)` -- a small three-variant line renderer
  ("item" plain, "sub" for "− label / −$val", "eq" for "= label / $val"
  with a top border + bold, mimicking a running subtotal).
- New `renderDispoBreakdown(prop, hold, d, dNo)` -- builds the full row set
  from `d` (post-offset dispoResults) and `dNo` (pre-offset snapshot, used
  ONLY to compute the "One-time obligation offset" line as a diff of two
  already-engine-computed `recognizedGain` values -- not a new calc).
  Rows conditionally appear by reading `prop.isPrimary` / `d.mode` exactly
  as specified: §121 exclusion primary-only; recapture/CA clawback/other-
  state-credit rental-only; cash boot partial_1031-only; deferred gain
  full_1031-or-partial_1031; CA-source carryforward rental-1031-only.
  Percentage labels (selling costs 6%, recapture 25%, fed cap gains 23.8%)
  read from `DISPO_DEFAULTS` live (a `dispoPct()` formatter), not hardcoded
  strings -- so a future Defaults-tab override of those rates keeps the
  labels honest.
- The existing compact "Proceeds summary (vs CPA sheet)" block (Sale
  price/Tax/Net, from v4.2.5) is now the collapsed-state header, per the
  user's explicit ask -- unchanged, just gained a toggle button
  (`▾ itemized breakdown` / `▲ hide breakdown`) next to it. New state
  `dispoBreakdownOpen` (object keyed by property id, default collapsed).
- New dev-mode `useEffect` (mirrors the pre-existing "obligation audit"
  pattern) warning if `netSale != gross-sellingCosts` or
  `afterTaxNetProceeds != preTax-totalTax` for ANY sold property (the
  pre-existing audit only covers obligation-year sellers) -- guards the
  itemized breakdown against silently drifting from disposeAsset's fields.

Bug caught by the new tests before this shipped (not by the pre-existing
suite): the dev-mode audit's `afterTax = preTax - totalTax` check doesn't
hold for `full_1031` -- that mode is a rolled swap, so `disposeAsset` hard-
codes `afterTaxNetProceeds = 0` regardless of the freed equity (`preTax`
computed as netSale-mortgagePayoff would be a large positive number for a
1031, e.g. the R13 test's first failure showed a ~$945K mismatch). The
PRE-EXISTING "obligation audit" `useEffect` already knew this (`if(d.mode
==='full_1031') continue;`) but I didn't carry that exception into the new
audit or the new R13 test on the first pass -- `run_tests.ps1` caught it
(R13 failed cleanly with the exact mismatch amount), fixed both the new
`useEffect` and R13 to skip full_1031 the same way, reran, 86/86 green.

New tests: R13 (engine-level, `window.__engine.buildScenario/makeParams`,
5 cases spanning sell/full_1031/partial_1031 across sixth/fifteenth/
barberry, asserts the two-equation reconciliation with the full_1031
exception) and R14 (UI-level: confirms no toggle exists at mode=keep,
confirms the sub-card is collapsed by default after selling, confirms
primary-only vs rental-only rows via `textContent()` on sixth vs
fifteenth, confirms the "1031 exchange" section + "Deferred gain (rolled)"
appear and "Cash boot" does NOT appear when fifteenth is switched to Full
1031 specifically).

--- TEST INFRASTRUCTURE NOTE (not a code bug, but cost real time) ---

First `run_tests.ps1` invocation this session: all 84 tests failed with
`net::ERR_CONNECTION_REFUSED`. Root cause: `run_tests.ps1`'s actual content
is just `npx playwright test retirement-simulator.spec.js --reporter=list
--headed` -- it does NOT start or stop the Vite dev server, contrary to
what CLAUDE.md's "Running tests" section currently says ("convenience
wrappers that also start/stop the Vite dev server"). No server was up on
5173 when it ran. Compounding this: earlier in the session I'd started and
manually killed several `npm run dev` instances for browser-driven
verification (Playwright screenshots via a throwaway script, not the
suite) -- two of those left orphaned `node` processes squatting on 5174/
5175 (Vite's fallback ports), which I found and killed via `netstat -ano` +
`taskkill`. Started a fresh single dev server, confirmed `curl` 200 on
5173, reran `run_tests.ps1` -- 84/84 passed (8.2m). CLAUDE.md's
"Running tests" section is stale on this point; flagged below, not fixed
this session (out of the scope the user asked for).

--- HOW TO PICK UP FROM HERE ---

1. Sale price vs. Net Worth appreciation is now a locked invariant (see
   CLAUDE.md "Property-centric schema (v4)" and the comments at both call
   sites in engine.js) -- if a future session is tempted to make
   disposition FMV grow with sale year again, that's very likely a
   regression, not a feature; confirm with the user first.
2. `prop.value` remains the ONLY per-property value input in the schema --
   there is still no separate "sale price" field. If the user ever wants
   sale price to diverge from the "Property value" slider (e.g. entering a
   different expected sale price than the current-value slider shows),
   that would need a genuinely new `hold.salePrice` field, not a tweak to
   the appreciation math.
3. `sliderTypeIn` (App.jsx, next to `slider`) is now the reusable click-to-
   type slider variant -- takes toText/fromText unit-conversion functions
   plus color/fontSize, and clamps to [min,max] on commit. Used by Property
   value and the LTR Monthly rent slider; other dollar sliders (STR $/
   night, MTR Rate/mo, HI debt balances, etc.) still use the plain `slider`
   helper and were not touched -- ask before assuming they should match.
4. Before starting a `run_tests.ps1` run, confirm something is actually
   listening on localhost:5173 (`curl -s -o /dev/null -w "%{http_code}" http
   ://localhost:5173`) -- the script will not start it for you (see backlog
   above), and stray dev-server processes from manual browser-verification
   sessions can silently occupy 5174/5175 instead if 5173 was already
   taken, masking the real problem.
5. `renderDispoBreakdown`/`dispoRow` (App.jsx, next to `sliderTypeIn`) are
   the itemized-breakdown building blocks -- display-only by design, reading
   `liveRows.dispoResults[id]` / `liveRows.dispoResultsNoOffset[id]`
   directly. If a future session adds a new disposeAsset return field or a
   new obligation-style post-hoc adjustment (like the same-year-all-sold
   tax bump, which the breakdown deliberately does NOT itemize -- see
   v4.2.6 section above), decide explicitly whether it needs its own row;
   don't assume it's already covered.
6. `full_1031` is a recurring special case across this codebase's tax
   reconciliation checks (pre-existing "obligation audit" useEffect, the
   new "dispo breakdown" useEffect, and R13): `afterTaxNetProceeds` is
   hard-coded to 0 for that mode (rolled into a replacement property, not
   cashed out), so `preTax - totalTax` does NOT equal it and any new
   audit/test touching this reconciliation formula needs the same
   `if(mode==='full_1031') continue/skip` -- caught once already this
   session (v4.2.5's audit had it, v4.2.6's initial pass didn't).
