=== SESSION 29 JOURNAL — v4.4.0 (Birth-Date Anchor, Per-Spouse SS, Medicare/FRA Derivation) ===

Date: 2026-07-10

Session goal: user asked for three things building on the v4.3.0 start-date
anchor work: (1) add real birth dates as the anchor for age-triggered events,
(2) give Social Security month-precision AND make it independently settable
per spouse (staggered claiming), (3) investigate and report scope BEFORE
touching Medicare/FRA (currently pegged to a hardcoded Nov-2026 date and a
year-only brendaFraYear), then derive both from birth date -- pausing to
flag before implementing if that rippled unexpectedly.

--- INVESTIGATION (before any code changed) ---

Read session27/28 journals first per house convention. Traced Medicare/FRA
pegging: healthMonthly() (engine.js) hardcoded Bob's Ericsson->Medicare
transition as calYear===2026&&calMonth<11 (Nov 2026), and Brenda's as a
year-only BASE.brendaMedYear(2032)/brendaFraYear(2034) check with no month.
~13 call sites referenced these across both engines + event timelines +
DEFAULTS_REGISTRY, none locked by an existing test (verified via grep) --
low regression risk to change the underlying dates.

Two real unknowns surfaced that couldn't be resolved from the code:
1. Actual birth dates -- BASE.myAge:65/wifeAge:60 (labels only, unused in any
   calc) and brendaMedYear/brendaFraYear (2032/2034, 65/67 exactly 2yr apart)
   pointed at inconsistent implied birth years for Brenda (1966 vs 1967).
   Asked the user directly rather than reverse-engineer a guess. Got: Bob
   Oct 18 1961, Brenda Jan 19 1967.
2. Bob's birthday (Oct 18) is NOT the 1st of the month, so the standard SSA
   "Medicare starts the month you turn 65" rule derives Oct 2026 -- one month
   EARLIER than the existing Nov-2026 hardcode. Flagged this explicitly
   before implementing (per the user's own instruction to pause on anything
   that moves an existing date) and asked whether Nov-2026 should be treated
   as a calibrated real fact (birthMonth+1, e.g. employer coverage lapsing
   Oct 31) or corrected. User confirmed: CORRECT to Oct 2026 -- the old
   Nov-2026 value was stale, not calibrated, and regression tests should lock
   the NEW corrected date, not the old one.
3. Confirmed scope for Brenda's SS: her benefit $ amount stays FLAT
   (BASE.brendaSsFRA) regardless of her chosen claim date -- no early/delayed
   formula exists for her today (unlike Bob's Early->FRA interpolation), and
   building one wasn't asked for; a deliberate scope decision, not an
   oversight, documented in CLAUDE.md for future sessions.

--- WHAT SHIPPED (v4.4.0) ---

engine.js:
- BASE gets four new birth-date fields: yourBirthYear:1961, yourBirthMonth:10
  (Bob), brendaBirthYear:1967, brendaBirthMonth:1 (Brenda) -- Defaults-tab
  editable, single source of truth for every age-triggered event.
- New deriveAgeAnchors(base): computes medicareYouYear/Month (birthYear+65),
  brendaMedYear/Month (birthYear+65), brendaFraYear/Month (birthYear+67) onto
  BASE in place. Runs once at module load and again at the END of
  applyDefaultsOverrides() (after overrides apply) -- so a stale saved
  override of one of the DERIVED fields (e.g. a pre-v4.4.0 brendaFraYear in a
  localStorage blob) gets overwritten right back to the birth-date-derived
  value. Verified via a dedicated test (U7) that a direct override attempt on
  brendaFraYear cannot stick.
- New ssClaimAge(startYear,startMonth,birthYear,birthMonth): one shared
  continuous-age formula, used by both the SS $-amount calc and the Simulator
  sidebar's derived age readout -- not duplicated three times.
- healthMonthly(): Bob's and Brenda's Ericsson->Medicare gates now compare an
  absolute calendar month (calYear*12+calMonth) against the derived
  medicareYouYear/Month and brendaMedYear/Month -- month-precision for both
  (Brenda's was year-only before). Bob's transition is now Oct 2026 (was
  Nov 2026) -- confirmed by the user as an intentional correction.
- buildScenario(): yourSs/brendaSs restructured from a flat per-row $/mo
  constant (year-only on/off check) into a genuine PERIOD DOLLAR TOTAL,
  summed month-by-month over the row's real months -- exactly mirroring the
  existing healthAnnual loop's monthOffset convention (no parallel logic
  invented). Every downstream read site adjusted to match: totalIncome/
  passive no longer multiply by monthsThisYear (yourSs/brendaSs are already
  period totals); estimateTax's call divides back down to a $/mo rate
  (preserving its existing, separately-pre-existing *12 approximation
  unchanged); row.push divides by monthsThisYear to recover the displayed
  $/mo rate, following the same "*12-in, /monthsThisYear-out" pattern v4.3.0
  established for other fields. The debt-availability mo-loop's SS component
  (_ss0, a row-level constant before) is now computed LIVE inside the loop
  using its own real calMonth1to12, so the HI-debt-paydown-pacing gauge sees
  the exact month a spouse's SS switches on instead of a year-level guess.
- Params gained ssStartMonth (Bob, new) and ssBrendaStartYear/ssBrendaStartMonth
  (Brenda, new) alongside the existing ssStartYear/ssAmount.

App.jsx:
- window.__engine gained ssClaimAge, deriveAgeAnchors, and healthMonthly
  (the last one was ALREADY imported/used internally but had never been
  exposed for Playwright -- a pre-existing gap this session's tests needed
  closed, not something v4.4.0 introduced).
- wfData's yourSsMo/brendaSsMo: both spouses now gate on an absolute calendar
  month (calYear*12+(d.getMonth()+1)) instead of a year-only check --
  already-monthly engine, so this is exact by construction, no proration
  math needed (unlike buildScenario above).
- wfData's "You -> Medicare" and "Brenda -> Medicare" event markers: replaced
  the hardcoded calYear===2026&&d.getMonth()===10 and the year-only
  brendaMedYear&&d.getMonth()===0 checks with BASE.medicareYouYear/Month and
  BASE.brendaMedYear/Month absolute-month comparisons.
- Annual-engine events timeline: the hardcoded `yr===2026` Medicare check
  became `yr===BASE.medicareYouYear`; "Your SS starts (age X)" and "Brenda SS
  spousal starts (FRA year)" labels (both already slightly wrong before this
  session -- they assumed claiming always happened at age 65 / at FRA) now
  read the real derived claiming age via yourSsAge/brendaSsAge.
- Simulator sidebar: replaced the "Your SS Start Age" 65-70 stepped toggle
  (Bob only, no control existed for Brenda at all) with two blocks -- "Your
  SS Start" and "Brenda SS Start" -- each a year slider (BASE.startYear..+20)
  + month slider (1-12) + a derived claiming-age readout ("2026 * Oct -> age
  65.0") + a non-blocking amber warning below age 62. Bob's SS $-amount
  formula generalized from stepped ssAge to continuous ssClaimAge (same
  Early->FRA interpolation rule, unchanged -- including the pre-existing
  quirk that ages past 67 don't actually get the delayed-credit bonus the old
  UI text implied; not fixed here, flagged in a comment).
- SS earnings-test card, its chart ref-lines, and the launch-header age
  display (BASE.myAge/wifeAge, which had drifted stale -- Bob is actually 64
  and Brenda 59 at the real Jul-2026 default start, not 65/60) all now read
  the derived age instead of the old static/stepped values.
- Defaults tab: new "Birth Dates" group (yourBirthYear/Month,
  brendaBirthYear/Month); removed "Brenda FRA year" and "Brenda -> Medicare
  (year)" from their old groups since both are now derived, not independently
  editable (editing them directly would silently revert on the next
  applyDefaultsOverrides call, which would have been confusing UX).
- SAVE_SCHEMA_VERSION bumped 4->5 (ssAge removed from the sc shape, no
  migration, per project convention) -- old pins are discarded, not migrated.

defaults.js: DEFAULTS and SC_DEFAULTS both updated -- ssAge replaced by
ssStartYear/ssStartMonth/ssBrendaStartYear/ssBrendaStartMonth, defaults
chosen to reproduce the OLD default scenario's behavior exactly (Bob claims
at his birth month in the year he turns 65; Brenda claims at her derived FRA
-- both defaults are now BASE.*BirthYear+65/67 expressions, not literals).

retirement-simulator.spec.js:

- E1 (SS age 65->67 sanity check) updated -- the old test clicked a "67"
  toggle button that no longer exists; now drags the new "Your SS Start Year"
  slider to 2028 (2yr later than the default 2026/Oct = age 65, same month =
  age 67).
- New Group U (11 tests): U1/U1b lock dual-engine SS-dollar agreement (Bob's
  partial-start-year proration, Brenda's full-year default-FRA claim); U2
  locks exact-month proration for an isolated custom claim date; U3/U3b lock
  ssClaimAge's math and the sidebar's rendered readout text; U4a-d lock the
  CORRECTED Medicare dates (2026/10, 2032/1, 2034/1 -- explicitly NOT the old
  2026/11) across BASE, the shared healthMonthly(), the monthly engine's
