--- NEVER-SELL A1-EXPOSURE CHECK ---

Built the check as a real counterfactual engine run (not a formula): extracted the pre-v5
engine.js from git HEAD into a scratchpad, wrote a patched copy of the CURRENT engine with ONLY
the A1 maintenance mechanism reverted to the old capped-reserve + `maintRedirect`-to-sweep

Result: A1's isolated effect on the never-sell scenario at yr20 is -$189K -- actually SMALLER

compound the redirect-years the way more years of INFLATED maintenance would, because the old
formula was capped and flat (uninflated) to begin with. maxDI unaffected (floor-driven in this
scenario, not maintenance-driven). User accepted this as clearing the concern.

--- THE RENTALOPCOST BUG (found via re-run, not by design) ---

Re-running after those fixes surfaced R1 still failing with a real (not precision-sized)
mismatch, and T3 failing on a real dollar gap too big to be rounding. Traced R1 precisely: old
engine's annual `rental` field was built via `unitSegmentNet()` -- already NET of rental
operating costs (STR platform/cleaning%, LTR vacancy%, etc). New engine's `rental` field is
GROSS (`avgMo('rental')`, sourced from wfData's own gross `rental`; `rentalOpCost` is a fully
separate field). Nothing else picked up the slack: `rentalOpCost` was in neither `tier1` nor
`totalOut` nor `passive`. It was just dropped.

Quantified against the real default scenario (which already has an active STR segment, so this
wasn't hypothetical): `reqWork` (the headline "Total Work Income Required" stat) was understated
by the full dropped op-cost amount every year -- ~$460-620/mo at launch, growing to make the
model claim "work-free" a year earlier than the true net-of-op-cost figure supports at yr10
($0 shown vs. the correct ~$1,131).

Presented this to the user with two fix options (net `rentalOpCost` out of `passive`, restoring
exact pre-v5 semantics; or make `rental` a documented gross line plus a new explicit `totalOut`
charge). User chose option 1 explicitly, ruling out option 2 as an unnecessary structure/behavior
change with double-count risk, out of scope for the v5 commit gate. Fixed:
`aggregateMonthlyToAnnual`'s `passive = avgMo('rental')+avgMo('pension')+avgMo('yourSs')+
avgMo('brendaSs') - avgMo('rentalOpCost')`. Verified the residual gap between old and new
`reqWork` at every sampled year is now fully explained by ALREADY-approved differences (B1's
debt-pace correction dominates, e.g. -$745/mo in `minDebt` at yr10, offset by small C1/C2/A1
precision effects of +$350/mo, netting to exactly the observed -$395 `totalOut` difference) --
decomposed component-by-component, not just asserted.

--- THE ZERO-VALUE BUG (found while building IRMAA/rentalOpCost's regression guards) ---

Added R16 (a direct engine-level IRMAA regression test) -- passed cleanly first try. Building
A5 into an actual regression guard (it previously asserted `rwZero <= rwBase`, trivially
satisfied even when nothing moved) surfaced something new: setting the "STR platform fee"
slider to 0% produced the EXACT SAME reqWork as the 3% default, to the dollar, in the real
browser. Traced it to App.jsx's `sc -> engine params` conversion:
`strPlatformPct:(sc.strPlatformPct||3)/100` -- `||` treats a literal 0 as falsy, so setting the
slider to 0% silently fell back to the 3% default. Confirmed via a standalone engine check with
the SAME buggy pattern reproduced (delta=0) and then fixed (delta=$84/mo, matching the real
expected 3% platform fee on 15th St's ~$33.6K/yr STR gross).

This is present at BOTH sc->params conversion sites in App.jsx (`buildRowsFromSnapshot` for
pins, and the live `liveParams` useMemo), for FOUR fields with the same `||nonzero-default`
pattern: `strPlatformPct`(->3), `strCleanPct`(->4), `ltrVacancyPct`(->4), `mtrCleaningFlat`
(->300). `mgrPct` is unaffected only because its default already is 0. Confirmed this predates
the v5 refactor entirely (the conversion lines are untouched by the single-engine collapse) --
a real, live v4.6.0 bug, found only because a regression-test rewrite happened to need a working
0% lever.

User's explicit decision: fix now (mechanical, same restore-correct-semantics pattern as the
other two), but as its OWN separately-versioned commit (v5.0.1), not folded into the v5.0.0
refactor story, since this one CHANGES v4.6.0 behavior rather than restoring a v5 regression.
Required checking the 3 real baseline pins first for safety -- confirmed all three use the
non-zero defaults for all 4 affected fields (strPlatformPct:3, strCleanPct:4, ltrVacancyPct:4,
mtrCleaningFlat:300 in every pin's `liveSc`), so the fix doesn't move any committed fixture.
Fixed: `||` -> `??` (nullish coalescing) at both call sites, for all 4 fields.

A5 (strengthened threshold + `setSliderExact('STR platform fee', ...)` instead of the fuzzy
`setSlider`, which was silently grabbing a wrong/unrelated slider under the real label mismatch)
now doubles as the regression guard for BOTH the rentalOpCost fix and the zero-value fix
simultaneously -- it can only pass if the slider genuinely reaches 0% (needs the v5.0.1 fix) AND
that 0% genuinely nets into reqWork (needs the v5.0.0 fix). A6 fixed the same way for consistency.

--- THE EXHAUSTIVE CHECKLIST AND THE FOURTH FINDING ---

User's bar for commit: three bugs found by audit (two of them the identical "old engine netted
a cost, new engine silently drops it" pattern) with a 114-test green suite catching none of
them cleanly is not sufficient assurance. Required a complete, positive-confirmation checklist:
every income line, cost, tax, surcharge, credit, and adjustment the old engine computed, mapped
to where it lands in the new engine, explicitly flagging anything computed differently or that
can't be fully accounted for.

Produced the full table (see below, "OLD -> NEW FIELD CHECKLIST" -- this is the artifact, kept
here verbatim for the record) covering every income line, every term of old's `baseOut`/new's
`tier1` (all nine now match, including the restored IRMAA term), the debt-sweep/one-time-pool-
routing mechanism, and every NW component. While tracing the one-time pool-year routing in
detail, found a fourth item: old's pool-year block had its OWN mortgage-principal-bucket step
(routing sale-proceeds residual through extra mortgage paydown, gated on `p.mtgPrincipalOn`,
BEFORE it could reach savings); new's pool-year block has no equivalent step -- only the ONGOING
per-month `mtgPrincipalOn` block exists.

Verified this precisely with a constructed scenario (sell 15th St 2026, `mtgPrincipalOn:true,
uncapped`, `payOffHI:true` to isolate the mechanism) and traced the TRUE resulting mortgage
balance one year out (not the same-year snapshot, which differs in timing convention between
the two engines' pool-year-vs-ordinary-month capture points and would have been a false alarm).
Old: 6th St ends at $229,424. New: ends at $278,718 (~$49K higher, i.e. less paid down). The
ongoing per-month `mtgPrincipalOn` block DOES catch the one-time residual (via `toSavings`'s
`_oneTimeToSavings` blend) -- this is NOT a case of money vanishing like rental/IRMAA. The ~$49K
gap is fully explained by the ALREADY-approved B2 fix: old's annual pool-year routing never
modeled reserve/buffer fills at all ("annual approximation: reserve buckets aren't modeled
here"), while new correctly fills rd/ob ($45,000 in this trace) before the mortgage bucket gets
a turn -- accounting for nearly all of the difference. User agreed this is the B2 fix working
correctly, not a bug.

A narrower, real issue DID surface from the same trace: the DISPLAYED `wfToSavings`/
`oneTimeSweep` field is captured from `_oneTimeSweep` at the top of the pool-year block, BEFORE
the ongoing `mtgPrincipalOn` block (which runs later in the same month) can reduce it via
`_takenFromSavings`. The actual money flow is correct -- `sweepToSavings`/`savingsAcc` (and
therefore NW) use the properly-reduced `toSavings` value -- but the displayed conservation-check
field (`settleDraw+wfDebtPaid+wfReserveFill+wfToSavings === residual`) overstates how much
reached true savings whenever `mtgPrincipalOn` is combined with a sale year, since it doesn't
account for the slice that actually went to mortgage paydown instead.

User's decision: do NOT fix in this commit. `mtgPrincipalOn` defaults off and is confirmed off
in all 3 baseline pins -- doesn't touch the committed baseline, and NW/reqWork/savingsAcc are
unaffected (only a display/conservation-check label is wrong in this narrow combination). Log
it as a named, diagnosed follow-up instead (see OPEN ITEMS below) so it isn't lost.
