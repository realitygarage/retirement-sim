--- 2. 6TH ST SEGMENTED INCOME EDITOR ---

Data shape (defaults.js, in SIXTH_STR_DEFAULTS so it lands in DEFAULTS +
SC_DEFAULTS + pin snapshots):
  sixthIncomeSegments: [
    { yrFrom, yrTo, kind: 'str'|'mtr'|'ltr',
      str: [{ days, rate, type:'nightly'|'monthly' }],   // Σ days <= 365/yr
      mtr: [{ months, rate }],                            // Σ months <= 12/yr
      ltr: { monthlyRent } }
  ]

Engine (engine.js):
- New exports: sixthSegmentGross(seg) -- annual gross for one outer segment,
  inner caps ENFORCED by clamping (excess days/months ignored);
  validateSixthSegments(segs) -- returns human-readable errors for overlapping
  outer year ranges and cap violations (UI display + tests).
- buildScenario 6th-income block consolidated: one sixthIncome value feeds
  both the annual loop and the monthly mirror (previously duplicated code --
  guarantees "empty segments == flat mode" identity).
- Precedence: non-empty sixthIncomeSegments overrides the simple mode selector
  (and sixthSTRStartYear/StopYear); empty list = simple mode + sixthMTR
  back-compat unchanged.
- Costs: STR-kind nets platform+cleaning+mgr % inline (like duplex-top STR);
  MTR/LTR kinds stay gross in the annual engine and net mgr % in the monthly
  wfData engine (same as the pre-existing 6th MTR treatment).
- rentGrowth applied via the same rg multiplier the duplex schedules use.
- sixthSTRStopOnDebtClear applies across ALL segment kinds (income continues
  through the clear year, stops the year after -- same semantics as STR).
- wfData (App.jsx) mirrors this: segment gross feeds the existing
  _sixthMtrMo/_sixthStrGrossMo variables so the existing rentalOpCost lines
  net the right knobs.

UI: "Income Segments" list inside the 6th St Income Mode card, same nested
pattern as the duplex strSchedule/mtrSchedule editors: outer rows with
From/To year sliders + STR/MTR/LTR kind toggle; inner editors per kind
(STR: days x nightly-or-monthly rate segs; MTR: months x rate blocks;
LTR: single rent slider). Cap counters go red over 365d/12mo; validator
errors render in a red box (data-testid="sixth-seg-errors"). When any
segment exists the simple-mode selector grays with a note.

--- 3. BUG FIXES ---

(a) Settlement card "Sale proceeds" showed $919K where ~$770K expected.
    Diagnosis (via node harness dumping the disposeAsset return object --
    the temporary console.log equivalent; not left in committed code):
    the card ALREADY used afterTaxNetProceeds and tax + mortgage payoff WERE
    subtracted. The real drop-out was upstream: computeDispo derived
    fmv = BASE value x (1+reApp)^yrIdx and silently IGNORED the disposition
    card's Sale price slider (d.salesPrice). A 2028 6th sale off 1.7M grown
    = $919K; off the $1,675K slider price = $787K-at-2026-level. Fix:
    fmv = def.salesPrice when > 0 (the slider IS the sale-year price; no
    appreciation growth applied to it), fallback to the old formula.
    NOTE: engine tax (~$49K, 23.8%+4.4% on post-§121 gain) still runs below
    the CPA sheet's $62K -- that residual delta shows in the reconciliation
    card and is a rate-calibration question for the CPA, not a plumbing bug.
(b) Reconciliation-vs-CPA rows now compare against a snapshot taken BEFORE
    the settlement gain-offset and same-year bump mutate the results
    (rows.dispoResultsNoOffset, shallow-cloned after the CA-cap pass).
    The CPA sheet assumes no offset, so the gainOffsetPct slider no longer
    moves the reconciliation deltas. Live totals (settlement panel, cash
    flow) still respond to the slider.

--- SEMANTIC NOTES ---

- Sale price slider is now the sale-YEAR price, not a 2026 figure grown by
  appreciation. Selling in a late year without touching the slider will
  understate proceeds -- drag the slider to your expected future price.
- Inner-cap enforcement is by clamping (excess days/months earn nothing),
  plus a visible validator error. Overlapping outer ranges: engine uses the
  first covering segment; validator flags it in red -- not silently merged.
- Segments override the START/STOP years of simple STR mode entirely; only
  sixthSTRStopOnDebtClear carries over (all kinds).
- The reconciliation card intentionally diverges from live totals when
  gainOffsetPct > 0 -- that is the point of fix (b).
