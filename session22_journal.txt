--- 1. PER-SCENARIO CHART LEGENDS (App.jsx, shared Chart component) ---

Investigated the bug report ("only the Free Cash Flow chart shows scenario
labels") by reading the shared `Chart` component (used by all 5 grid charts:
Total Work Income Required, Free Cash Flow/mo, HI Debt Balance, Net Worth,
Fixed Costs/mo). Finding: NONE of them actually rendered a per-pin legend --
each `<Line>` had a `name={pin.name}` prop but there was no `<Recharts Legend>`
or custom label anywhere in the component, on ANY of the 5 charts. (The
separate "Cumulative Income vs Cost" chart, built independently below the
grid, does have a working Recharts `<Legend>` with pin names -- that one was
already fine and untouched.)

Fix: added a small custom legend row (swatch + name, same visual style as the
existing secondary/tertiary "Free Cash / -> Swept" header indicators) to the
shared `Chart` component, right after the header and before the
ResponsiveContainer. Renders one entry per visible pin plus "Live" when
showLive is on. Since all 5 grid charts share this one component, this fixes
all of them uniformly in one place.

Bug caught during testing: the legend entries' outer wrapper was a `<div>`
whose only text content was the pin's exact name (svg swatch contributes no
text). This collided with an EXISTING test (D4) that does
`page.locator('div').filter({hasText: /^PinName$/})` to find the pin card in
the side panel -- with 5 new divs now also matching that exact-text pattern,
`.first()` grabbed one of the new legend divs instead of the real pin card,
and the test's downstream traversal broke. Fixed by using `<span>` instead of
`<div>` for the legend row and each per-pin entry (flex layout works the same
on span) -- Playwright's `locator('div')` no longer matches them. Re-ran D4 to
confirm the fix; all other groups unaffected.

--- 2. PER-SCENARIO PIN COLORS (App.jsx) ---

pin.color already existed and already drove every chart's line color (set at
creation time from PIN_COLORS[nextId % PIN_COLORS.length] -- a 6-color
palette in defaults.js, already "sensible defaults for new pins" per spec).
What was missing was any UI to CHANGE it after the fact.

Added: a `setPinColor(id, color)` callback (mirrors removePin's shape --
map/replace, setPins, persist via savePinsToStorage) and a small native
`<input type="color">` swatch on each pinned-scenario card (Pinned Scenarios
panel), wired to it. Because every chart's Line and the new legend both read
pin.color directly and reactively, changing the swatch immediately repaints
the line + legend on every comparison chart with no further plumbing needed.

--- 3. FREE CASH FLOW CHART EXCLUDES THE ONE-TIME DRAW (App.jsx, wfData) ---

Traced the FCF/mo chart's series: dataKey="surplus" -> chartData.surplus ->
discByYear (annual avg of wfData's monthly `disc` field). Found the root
cause at the `disc` computation in the monthly waterfall loop: `disc = (...)
+ _settleDrawMo` -- the one-time settlement draw was being added into the
"what you keep as recurring Free Cash Flow" figure every month it occurred,
spiking that month's average and blowing out the chart's Y-axis scale
(surplusMax derives from max(chartData.surplus), so the whole chart's scale
stretched to fit one lump-sum month).

Fix: removed `+ _settleDrawMo` from `disc`. Confirmed via code trace that
`disc` is a pure REPORTING value only -- it is pushed into the row output and
consumed by chartData/the Month-by-Month table's "Free Cash" column, but
never feeds back into ccBal/sophiaBal/nolanBal/savingsAcc or any other state
carried to the next month's iteration -- so this is a display-only change,
zero engine/waterfall impact. The draw itself is UNCHANGED in the pool ->
obligation -> draw -> cascade routing (still leaves the pool via
`split.draw` before planHiPaydown runs) and remains fully visible via:
row.settleDraw (unchanged), its own event marker ("Obligation-year one-time
draw $XXXK", unchanged), and the Cash-Flow Engine routing display (unchanged,
reads rSettle.settleDraw from the annual engine separately).

Note: engine.js's OWN annual "Free Cash Flow -- breakdown" table (liveRows,
used only for the FCF chart's expandable breakdown panel) still includes
drawInc inside its surplus/totalIncome, with drawInc also shown as its own
labelled line item ("Lifestyle Draw"). Left this untouched -- it's the
annual engine (out of scope per user's instruction), and unlike the monthly
chart series it's an itemized accounting breakdown, not a single line that
visually spikes.

--- BUGS CAUGHT DURING TEST-WRITING (not app bugs, test-authoring pitfalls) ---

- setSlider's proximity-based (`filter({near: ...})`) lookup grabbed the WRONG
  control for "One-time draw ($, at sale)" -- it silently matched a property
  VALUE slider (min/max 837500/3350000) instead, because that control sits
  far down a long sidebar and the heuristic isn't reliable at that distance.
  No error was thrown (a slider was found, just the wrong one), so this
  produced a confusing "draw amount stayed 0" failure rather than an obvious
  test-tooling error. Root-caused by dumping the located slider's min/max
  before asserting anything else. Fixed by walking the actual DOM structure
  (label span -> ancestor::div[2] -> sibling input[type=range]) instead of
  relying on visual proximity -- deterministic regardless of scroll position
  or sidebar length.
- The D4 pre-existing-test regression above (legend divs colliding with
  exact-text pin-name div lookups) -- see section 1.
