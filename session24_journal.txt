--- 1. CONFIRMED THE DIAGNOSIS: TWO SEPARATE CODE PATHS (App.jsx) ---

Live path: the `wfData` monthly waterfall block (App.jsx ~507-548) tracks the
one-time settlement draw (`_settleDrawMo`) entirely separately from `totalInc`
-- it is never added to monthly income, so `disc` (~732-734) never sees it.
The "exclusion" is really "never included in the first place."

Pinned path: `addPin` (~1029) stores `rows = buildScenario(liveParams)` --
pins only ever run the ANNUAL engine, never the monthly wfData block. The
annual engine (engine.js ~806-807) explicitly adds the draw into
`totalIncome` as `drawInc`, which flows straight into `surplus`. The FCF
chart's pinned series (`pin_<id>_di`, App.jsx ~928) read
`Math.max(0, pin.rows[i]?.surplus)` directly -- leaking the draw as a spike.

Fix: added a shared helper in the `chartData` memo --
`annualFcfExDraw(row) = Math.max(0, row.surplus - (row.settleDraw||0))` --
used both for the pinned series and for live's own no-wfData fallback path
(the `Math.max(0,r.surplus)` branch that only fires if `discByYear[r.cal]` is
null), so the two paths can't drift apart on this again.

--- 2. VERIFICATION FOUND A SECOND, LARGER, PRE-EXISTING BUG (not fixed) ---

Before writing the closing test, drove the real app (sell 6th St 2027, later
also a $100K one-time draw on 15th St 2026) and diffed live vs. pinned FCF
values year-by-year via window.__wfData/__liveRows. The draw-leak mechanism
checked out, but the pinned line was also wrong by ~3x in EVERY year after HI
debt clears -- not just the draw year, and independent of any draw at all
(confirmed with settleLifestyleDraw=0).

Root cause (engine.js ~955-962 vs App.jsx ~732-734): once HI debt clears, the
annual engine reports the FULL disposable income (`baseDI`) as `surplus`,
while the monthly engine reports only `lifestyleSplit%` of it as `disc` (30%
default -- the rest sweeps to savings by design). These two quantities happen
to coincide while debt is still active (both effectively floor/split-
protected) and permanently diverge once it clears. This is the same "annual
vs monthly engines disagree" class of issue already flagged as open P1 in
session 23 (there: HI-debt-clear TIMING; here: the FCF VALUE itself, once
cleared) -- likely the same underlying architecture gap, not yet unified.

Flagged this to the user before proceeding rather than silently writing a
test that overclaims live/pinned equality. User's call: ship the draw-leak
fix only (as originally scoped -- "chart attribution only, don't touch pool
math"), narrow the closing test to the draw-leak contract specifically, and
log the post-debt-clear divergence as new backlog (see below) rather than
attempt a fix this session.
