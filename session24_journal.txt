--- 1. CONFIRMED THE DIAGNOSIS: TWO SEPARATE CODE PATHS (App.jsx) ---

Live path: the `wfData` monthly waterfall block (App.jsx ~507-548) tracks the
one-time settlement draw (`_settleDrawMo`) entirely separately from `totalInc`
-- it is never added to monthly income, so `disc` (~732-734) never sees it.
The "exclusion" is really "never included in the first place."

Pinned path: `addPin` (~1029) stores `rows = buildScenario(liveParams)` --
pins only ever run the ANNUAL engine, never the monthly wfData block. The
annual engine (engine.js ~806-807) explicitly adds the draw into
`totalIncome` as `drawInc`, which flows straight into `surplus`. The FCF
chart's pinned series (`pin_<id>_di`, App.jsx ~928) read
`Math.max(0, pin.rows[i]?.surplus)` directly -- leaking the draw as a spike.

v4.1.4 fix: added a helper in the `chartData` memo --
`annualFcfExDraw(row) = Math.max(0, row.surplus - (row.settleDraw||0))` --
used both for the pinned series and for live's own no-wfData fallback path,
so the two paths can't drift apart on this again. (Superseded by v4.1.5,
section 4 -- the helper's body changed but its call sites didn't.)

--- 2. VERIFICATION FOUND A SECOND, LARGER, PRE-EXISTING BUG ---

Before writing the closing test, drove the real app (sell 6th St 2027, later
also a $100K one-time draw on 15th St 2026) and diffed live vs. pinned FCF
values year-by-year via window.__wfData/__liveRows. The draw-leak mechanism
checked out, but the pinned line was also wrong by ~3x in EVERY year after HI
debt clears -- not just the draw year, and independent of any draw at all
(confirmed with settleLifestyleDraw=0).

Root cause (engine.js ~955-962 vs App.jsx ~732-734): once HI debt clears, the
annual engine reports the FULL disposable income (`baseDI`) as `surplus`,
while the monthly engine reports only `lifestyleSplit%` of it as `disc` (30%
default -- the rest sweeps to savings by design). These two quantities happen
to coincide while debt is still active (both effectively floor/split-
protected) and permanently diverge once it clears.

Flagged this to the user before proceeding rather than silently writing a
test that overclaims live/pinned equality. Shipped v4.1.4 (draw-leak fix
only) first, with the divergence logged as new backlog and a narrowly-scoped
closing test (S6) that deliberately did NOT compare pinned vs. live directly.

--- 4. v4.1.5 FIX: engine.js FIELD, NOT AN App.jsx PATCH ---

Traced why `surplus` can't just be edited in place: `surplus` fields into
`reqWork` and `nw` elsewhere in the same function (engine.js ~961-965) --
changing its formula risks moving numbers used outside the FCF chart. Also
the App.jsx-side "subtract settleDraw from surplus" approach from v4.1.4 is
only a crude approximation of what the monthly engine actually does (the
draw never touches `totalIncome`, so its ACTUAL effect on `disc` scales with
whatever fraction of disposable income currently flows through
lifestyleSplit%/floor, not a flat 1:1 subtraction) -- confirmed empirically:
a $100K draw only moved the affected year's value by ~$2,700, not ~$100K.

Fix: added ONE new, purely-additive, read-only field to buildScenario()'s row
output -- `fcfChart` (engine.js ~962-975) -- computed as:
  baseDIExDraw       = baseDI - drawByYear[cal]           (excl. the draw)
  splitProtectExDraw = max(diCap*12, baseDIExDraw*lifestyleSplit%)
  fcfChart           = max(0, min(baseDIExDraw, splitProtectExDraw))
This reduces to the EXACT pre-existing `surplus` value whenever there's no
draw and debt is still active (min(baseDI,splitProtect) is algebraically what
`surplus` already equals in that branch -- see the derivation in code
comments), and correctly applies the split-protected "kept" amount once debt
clears (the actual fix), for ANY debt state, with the draw cleanly excluded
throughout. `surplus`, `accel`, `reqWork`, `nw`, `totalOut` are completely
untouched -- `fcfChart` is a new sibling field, not a replacement.

App.jsx: `annualFcfExDraw()`'s body now just returns `row.fcfChart` (both
call sites -- the pinned series and live's own no-wfData fallback --
unchanged, so this is a one-line implementation swap, not a call-site
rewrite).

--- 6. USER FOUND A RELATED GAP v4.1.5 MISSED: THE PINNED SWEEP LINE (v4.1.6) ---

After v4.1.5 shipped, user pinned a fresh "easy st - 1" scenario with NO
changes to live and spotted (via screenshot) that the FCF/di line was much
closer now, but the chart's secondary "Sweep" series (blue dashed) was
wildly off -- pinned $9,317/mo vs. live's $1,564/mo in a routine, non-sale,
non-draw year. Root cause: `pin_<id>_sweep` (App.jsx ~930) still read the
raw `debtSweep + sweepToSavings` fields directly, which is exactly the
pre-fcfChart pattern v4.1.5 replaced for the FCF line -- `sweepToSavings`
uses `baseDI` (not draw-excluded) and only turns on post-clear, inheriting
both bugs v4.1.5 fixed for `surplus`/FCF.

Fix: added `sweepChart = max(0, baseDIExDraw - fcfChart)` to
buildScenario()'s row output -- the direct algebraic complement of
`fcfChart`, purely additive (doesn't touch `debtSweep`/`sweepToSavings`,
which are still correctly used elsewhere for pin NW/savings accumulation,
where including the real draw amount is desired). Swapped
`pin_<id>_sweep` to read `row.sweepChart`.

--- 7. STILL OFF -- FOUND THE REAL ROOT CAUSE ONE LEVEL DOWN (v4.1.7) ---

v4.1.6 improved the Sweep line (9,317 -> 360 vs. live's 1,564) but flipped
it to badly UNDER-shooting instead, and the FCF line still didn't match
either ($1,700 pinned vs. $800 live, same identical-params pin). Traced by
reconstructing the arithmetic from the tooltip values rather than guessing:
fcfChart(1700) + sweepChart(360) = baseDIExDraw(~2060/mo), and the annual
model's `splitProtectExDraw` was hitting its floor branch at exactly 1700 --
which is `discFloor(800) + rdTopUp(400) + obTopUp(500)` = `diCap`, the app's
default values, confirming `fcfChart`/`sweepChart` were using `p.diCap` as
the floor. But the monthly engine's own floor comparison (`cfSplitProtect`
in the wfData block) uses `effectiveFloor` = bare `discFloor` (800) alone --
rdTopUp/obTopUp are already subtracted from `available` earlier in the
monthly waterfall as their own tiers, so folding them into the floor AGAIN
via `diCap` double-counted them, inflating the annual model's "kept" amount
by exactly the rd/ob top-up sum and starving the swept amount by the same.

Fix: `splitProtectExDraw` now uses `p.discFloor*12` instead of `p.diCap*12`.
Also added `discFloor: 800` to `DEFAULTS` in defaults.js (it previously only
existed in the app-facing `SC_DEFAULTS`) so direct `buildScenario(makeParams
({...}))` calls -- used by several existing tests that bypass the `sc`
spread -- don't get `NaN` from `p.discFloor` being undefined.

Result after v4.1.7: user confirmed "mostly fix, still a bit of difference,
but only a few hundred dollars" -- explicitly deferred to a later
refactor rather than chasing further now (see Open Backlog). The likely
remaining cause is the deeper, pre-existing annual-vs-monthly total
disagreement noted in the P1 "Unify annual sweep model with wfData" item:
the annual model's `baseDI` doesn't subtract the ongoing monthly rd/ob
top-up *contributions* the way the monthly engine's `available` does (only
the floor-comparison threshold was fixed here, not the income total feeding
into it) -- this was NOT chased further per the user's explicit call.
