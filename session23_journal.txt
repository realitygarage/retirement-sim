--- 1. TOOLTIP LABELS BLANK ON NON-FCF CHARTS (v4.1.1, App.jsx) ---

User's screenshots showed the v4.1.0 per-pin legend rendering correctly (pin
names + colors visible at the top of Net Worth, HI Debt Balance, Fixed Costs)
but hovering a point showed tooltip rows like ": $61" with no name before the
colon -- only on the non-FCF charts.

Root cause: the shared `Chart` component's `<Tooltip formatter>` had
`if(!secondaryDataKey) return [fmtV, ""];` as its first line. `secondaryDataKey`
is only ever set on the FCF/mo chart (the "Sweep" secondary line) -- so every
other chart (Net Worth, HI Debt, Fixed Costs, Work Income Required) hit that
line and got a hardcoded blank name for every series, Live or pinned, on
every hover, regardless of the v4.1.0 legend fix.

Fix: removed the blanket early return. The secondary/tertiary/"Free Cash"
rename logic now only runs when `secondaryDataKey` exists (still FCF-only,
unchanged behavior there); every other chart falls through to `return [fmtV,
name]`, i.e. the actual `name` prop already set on each `<Line>` (`"Live"` or
`pin.name`) -- the same value the legend already displays.

New test: S4 -- pins a scenario named "Tooltip Test", hovers the Net Worth
chart (the one from the bug report), reads every
`.recharts-tooltip-item-name` element, asserts none are blank and that both
"Live" and "Tooltip Test" appear.

--- 2. "DEBT CLEAR" MARKER NOT TRACKING THE HI DEBT BALANCE CHART (v4.1.2) ---

User's scenario: selling 6th St + paying down HI debt, deciding on SS start
age. Reported two things:
  (a) SS age 65->80 didn't move the FCF/mo chart at all.
  (b) The FCF chart's "debt clear" vertical reference line didn't line up
      with where the HI Debt Balance chart's line visually hit zero, and
      moved in ways that didn't track that chart.

Investigated (a) by driving both engines directly via a throwaway Playwright
script against the running dev server (window.__wfData/__liveRows), rather
than guessing from source:
  - Confirmed disc (FCF/mo) is genuinely floor-clamped and IDENTICAL between
    SS=65 and SS=70 while HI debt still has a balance -- everything above the
    floor sweeps to debt regardless of SS timing, so disc pins at the floor
    either way. This is by design, not a bug.
  - Confirmed reqWork (Total Work Income Required) DOES move a lot with SS
    age (e.g. $5,961/mo -> $8,498/mo in 2026 for SS 65 vs 70 in one test run)
    -- required work income auto-compensates for delayed SS to keep the same
    surplus/floor target, which is *why* FCF/mo doesn't move: the visible
    effect of SS timing while debt is active is on the Work Income Required
    chart, not FCF/mo. Reported this back to the user as expected behavior,
    no code change.

Investigated (b) and found a real bug: `debtClearYear` (App.jsx) was computed
as "first year chartData.sweepToSavings > 0" -- but that's the year the
surplus sweep REDIRECTS to savings, which lags the actual debt-zero moment by
the "Grace period (whoop it up first)" slider (sweepDelay, 0-24 months,
App.jsx ~3101) PLUS, in some parameter combinations, by the actual (separate)
HI Debt Balance chart plotting the ANNUAL engine's hiDebt series while the old
debtClearYear proxy was sourced from the MONTHLY engine's sweepToSavings --
two different signals from two different engines, wearing the same label.

Fix (per user's choice: "Both -- two markers", after I flagged the tradeoffs):
  - `debtClearYear` now = first year `chartData.hiDebt <= 0` -- same series
    the HI Debt Balance chart plots (annual engine, liveRows). Guaranteed to
    align with that chart's own zero-crossing. Colored green (matches HI Debt
    Balance chart's own $0 reference line).
  - New `sweepToSavingsYear` = the OLD proxy computation, kept as-is but
    correctly labeled "sweep -> savings" now that it's not masquerading as
    debt-clear. Colored blue (matches the FCF chart's existing "Sweep"
    legend color). Only rendered when it differs from debtClearYear.
  - Exposed both as `window.__chartMarkers` (same test-scaffolding pattern as
    __wfData/__liveRows) so tests can assert on the real computed values
    instead of scraping chart pixel positions.

IMPORTANT FINDING surfaced while stress-testing this fix, NOT fixed this
session (user's explicit call: "ship the marker fix, defer this"):

The annual engine (engine.js buildScenario, which drives liveRows / the HI
Debt Balance chart) and the monthly engine (App.jsx wfData block, the
doc-designated authoritative engine for HI-debt avalanche paydown) DISAGREE
on when HI debt actually reaches zero -- by 2 years in one no-property-sale
test case (annual: 2036, monthly: 2034, with sweepDelay=0 so grace period
isn't the explanation). Both engines run their own month-by-month avalanche
loop internally (annual engine's is at engine.js ~855-895, inside
buildScenario's per-year loop; monthly engine's is the wfData block's own
per-month avalanche) and they are not producing the same zero-crossing month
for the same inputs -- a genuine violation of the "two engines must never
disagree" invariant documented in CLAUDE.md, predating this session (not
introduced by the v4.1.0/4.1.1/4.1.2 chart work).

Practical consequence of shipping the "two markers" fix as-is: the two
markers can render in a visually confusing order (a "sweep -> savings"
marker appearing to the LEFT of "debt clear" on the calendar-year axis) in
scenarios where the grace-period offset doesn't happen to counteract the
annual/monthly gap. Confirmed via window.__chartMarkers with sweepDelay=0:
sweepToSavingsYear=2034 < debtClearYear=2036. This is surfaced, not hidden --
it's a real property of the underlying engine disagreement, and my fix
correctly reflects each engine's own truth rather than picking a value that
lies about one of them.

New test: S5 -- confirms debtClearYear matches window.__liveRows' own hiDebt
zero-crossing (the actual regression target), confirms both DOM labels
render distinctly whenever the two computed years differ (true by default in
this codebase's current param set: 2034 vs 2036), and re-checks the same
contract holds after changing the grace-period slider.

--- 4. ANNUAL/MONTHLY HI-DEBT-CLEAR DIVERGENCE: INVESTIGATED (v4.1.3) ---

User asked to look into the divergence deferred in section 2. First finding
was a CORRECTION to that section's own numbers, not a new app bug:

CORRECTION to section 2's "~2 year gap (2036 vs 2034)": that comparison came
from a throwaway diagnostic script (not committed, scratchpad-only) that
assumed wfData's month index starts at January 2026. It actually starts at
`new Date(2026, 5)` -- JUNE 2026 (App.jsx ~489, matching this session's real-
world "today"). Re-running the comparison with the correct month mapping
showed mortgage balances (primBalRaw/dplxBalRaw vs mtgBal6/mtgBal15) match
EXACTLY, to the dollar, every year -- the mortgage recast timing and amount
are byte-identical between engines; that part of the architecture is fine.
The REAL remaining gap, once measured correctly, is ~1 year (HI debt clears
2035 in wfData vs 2036 in the annual engine), not 2.

Root-caused the real ~1yr gap: buildScenario()'s internal per-month
sub-loop LOOKS like a true monthly simulation but several inputs
(_hlth0/_work0/_ss0/_rental0) are computed ONCE PER YEAR and held flat across
all 12 iterations, unlike wfData which recomputes exact per-month values
(including exact mid-year transition timing). Health specifically had a
concrete, confirmable bug: the shared `healthMonthly(calYear, calMonth, p)`
compares `calMonth<5` for the You->Medicare Ericsson cutover, but engine.js's
two call sites BOTH passed a sentinel `calMonth=99` (dead code -- the
branch was never actually exercised), while a separate hardcoded
`healthAnnual` yr===0 special case used `5*Ericsson+7*Medicare` (implying a
JUNE 2026 transition), and wfData's own separately-maintained inline health
calc used `mo<5` relative to ITS Jun-2026 epoch (correctly landing on NOVEMBER
2026, matching the on-screen Financial Events Timeline's "(Nov 2026)" label).
Three different implementations, three different implied transition dates
(May-ish dead code / June hardcoded / November actual), none of them talking
to each other. Confirmed with the user that November 2026 is the real date
before touching any code (this is a real fact about the household's health
insurance, not an arbitrary constant).

Fix (user chose "fix health-cost duplication first," the narrowest of three
options offered -- the other two were "leave it, treat wfData as
authoritative" and "full parity refactor of work/rental/SS too"):
  - `healthMonthly()`'s "you" branch corrected to the real Nov 2026 cutover
    (calYear<2026 || (calYear===2026 && calMonth<11) -- Ericsson; else
    Medicare).
  - engine.js's per-month loop now calls `healthMonthly(cal, mo+1, p)` for
    the ACTUAL month each iteration (was a flat `_hlth0` computed once,
    outside the loop, via the sentinel).
  - `healthAnnual`'s hardcoded yr===0 special case removed; now sums 12 real
    `healthMonthly(cal, hm, p)` calls, which naturally produces the correct
    10mo-Ericsson/2mo-Medicare split for 2026 without a special case.
  - App.jsx's wfData block's duplicated inline health calc (hiMo/brendaHlth/
    kidsHlth, ~5 lines) removed; now calls the same shared `healthMonthly()`.
    healthMonthly needed adding to App.jsx's engine.js import list.

Verified impact: all 77 tests still pass. Numbers moved in the expected
direction (e.g. default-scenario reqWork $5,961/mo -> $6,171/mo; the 2027
timeline event now reads "Health insurance drops $2,008 -> $1,618/mo" instead
of the previously understated "$1,798 -> ..." -- the annual engine had been
using the cheaper Medicare rate too early in 2026).

IMPORTANT: re-checked the annual-vs-monthly hiDebt trajectory after the fix,
same test scenario as before -- BYTE-IDENTICAL to pre-fix (still 2035 vs
2036). The health-cost fix is correct and worth having (real duplicated-logic
risk removed, real wrong date corrected) but did NOT move this specific
divergence at all: the annual engine's own `debtSweep`/avalanche-extra field
was already $0 for years 2028-2034 straight in this scenario (confirmed via
row inspection) -- its HI debt declines purely from minimum payments,
unaffected by a few hundred dollars/month of health cost either way, because
`_avail` never crosses the `_splitProtect` threshold regardless. So the real
driver of the ~1yr gap is the SAME once-per-year-flattening pattern applied
to `_work0`/`_ss0`/`_rental0` (not touched this round -- out of the
"health-cost duplication only" scope the user approved).
